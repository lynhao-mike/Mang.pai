## 归档清单

| 版本 | 文件 | 归档原因 |

| v5.0 | `blind-bazi-analyst-v5.0.md` | 单体架构 (~600+ 行) 触发 KIRO 平台超时；已被 v6.0 模块化架构替代 |

## 为什么要归档？

v5.0 把"理论框架 / 排盘流程 / 体用做功 / 流年大运 / 输出格式 / 自我迭代协议 / 神煞白名单"全部写进同一个 skill 文件，并通过 `#[[file:]]` 一次性拉入多份大型 PDF 与全部案例文件，导致：

1. **单次推理上下文过载** —— Claude 4.5/4.6/4.7 在长上下文+复杂指令场景下容易触发平台 timeout/OOM。
2. **任务边界不清** —— "排盘"和"应期推演"耦合在同一个 prompt 里，模型必须一次性算完，无法分段输出。
3. **无法接受用户反馈** —— v5.0 流程是 12 步直出，没有"暂停—等反馈—续推"的能力，命主修正的事实无法回灌。

## v6.0 修复策略

- 主入口 skill 仅做**路由 + 强制三步交互**，不内嵌理论。
- 理论按"排盘/格局/运气"拆为三个独立模块文件，按需 include。
- 每一步输出后**显式 STOP**，等待用户回反馈再进入下一阶段。
