# META · 自动入库协议（Ingestion Protocol · v1.0）

> **本文件版本**：v1.0（独立于主 Skill v6.1.0，META 层属于基础设施层，单独走版本号）
> **首发日期**：2026-05-18（FEAT-005）
> **维护方**：blind-bazi-analyst skill（自我迭代基础设施四件套之一）
> **角色**：把"用户往仓库根目录或 `案例库/` 丢一份新 md / PDF"这件事，从"靠 Kiro 主人手工搬"升级为"按协议自主走完识别 → 分类 → 提取 → 候选入库 → 晋级回写"的五阶段流水线。

> **协同伙伴**（META 四件套 · 互相闭环）：
> - `META/module-coverage-matrix.md` —— 全量映射『哪份文档已被哪些模块吸收』，每次入库后必须重算
> - `META/pending-extraction-queue.md` —— 已识别但尚未提炼为规律的文档片段或文档整体
> - `META/source-trace.md` —— 每条规律的『来源文档 + 来源段落 + 当前模块映射』三元组（audit log）

---

## §I.0 触发场景（什么时候执行本协议）

本协议由以下 4 类信号触发，按优先级从高到低排列：

| 优先级 | 触发信号 | 触发动作 |
|---|---|---|
| P0 · 强 | 用户**显式**说『把 XX 文档融入 Skill』『阅读这份 PDF』『学习这份案例』 | 立即从阶段 1 走到阶段 4；阶段 5 等用户授权 |
| P1 · 中 | 用户在仓库根目录或 `案例库/` 下**新增 md / PDF**（git status 显示 untracked） | Skill 在下一次被加载时（`blind-bazi-analyst.md §0` 启动自检）告警 |
| P2 · 弱 | 用户**修改了**已入库的 md（git status 显示 modified） | 仅对 `module-coverage-matrix.md` 中已 ☑ 的条目做差异扫描 |
| P3 · 周期 | 主 Skill 发版（v6.1.0 → v6.2.0 等）的 release window | 强制全量重跑阶段 1，校验 module-coverage-matrix 是否漂移 |

> **铁律 §I.0-A**：本协议**不阻塞用户问命**。
> 阶段 1（识别）跑完后若发现新文件，仅在 Step 1 反馈请求块**前**追加一行『⚠ 检测到 X 份新文档未入库，是否触发 META 协议？』，**等用户授权**才进入阶段 2-5。
> 用户拒绝 / 没回答 → 待办留在 `pending-extraction-queue.md`，下轮再问。

> **铁律 §I.0-B**：阶段 2-5 涉及**写入**主模块文件 / `cases/<专题>.md` / `blind-bazi-cases-index.md`，**必须**人审 + 至少 1 个新案例验证（详见 §I.5）。

## §I.1 五阶段协议（核心）

```
阶段 1 识别 → 阶段 2 分类 → 阶段 3 提取 → 阶段 4 候选入库 → 阶段 5 晋级回写
   ↑                                                                     ↓
   └─────────────── 阶段 5 完成后回到阶段 1，重算 module-coverage-matrix ──┘
```

### §I.1.1 阶段 1 · 识别（自动）

**目标**：扫描根目录 + `案例库/` 下所有 md / PDF，输出『新增 / 未入库 / 修改 / 已入库』四态清单。

| Step | 动作 | 工具 |
|---|---|---|
| 1-1 | `glob` 列出根目录 + `案例库/` 全部 `*.md` / `*.pdf` | `file_search` 或 `read_directory` |
| 1-2 | 与 `META/module-coverage-matrix.md` 现有行做 diff | grep + 字符串比对 |
| 1-3 | 输出四态清单：新增 / 修改 / 未入库（已存在但矩阵中显示 ☐）/ 已入库 | 写入临时区，**不**直接写矩阵 |
| 1-4 | 若新增 ≥ 1 份 → 在 `blind-bazi-analyst.md §0` 启动自检中追加告警 | 非破坏性 |

**输出格式（阶段 1 终态）**：

```
扫描时间：YYYY-MM-DD HH:MM
新增文件：N 份
  - 案例库/案例004_XX.md（未入矩阵）
  - 案例库/案例005_XX.md（未入矩阵）
修改文件：M 份
  - .kiro/skills/modules/module-c-yunqi.md（与矩阵列不一致）
未入库文件（已知 ☐）：K 份
  - 高德臣-盲派命理2018弟子班《理法篇》讲义31页_compressed.pdf
  - …（共 9 份 PDF）
已入库文件（已知 ☑ / ◐）：L 份
```

### §I.1.2 阶段 2 · 分类（自动）

**目标**：对阶段 1 输出的『新增 + 修改』文件做类型分类，决定走哪条提取分支。

| 类别 | 识别特征 | 提取分支 |
|---|---|---|
| **理论提示词** | 文件名含 `提示词_v*` / 内容首段含『System Prompt』 | §I.1.3 分支 A：抓铁律 / 表 / 编号 |
| **案例验证** | 路径在 `案例库/` 下 / 文件名前缀为 `案例NNN_` | §I.1.3 分支 B：抓 ADD / REVISE / VERIFY |
| **合婚报告** | 文件名含 `合婚分析报告` | §I.1.3 分支 C：抓 10 维评分 + 三窗口 |
| **择日报告** | 文件名含 `剖腹产` / `择日` / `命书` | §I.1.3 分支 D：抓七层递进 + 三体系投票 |
| **高德臣讲义 PDF** | 文件名含 `高德臣` 且为 `*.pdf` | §I.1.3 分支 E：走 §I.2 PDF 子协议 |
| **其它** | 不符合上述 5 类 | 标 N/A 或 ☐ 待人工确认 |

> **铁律 §I.1.2-A**：分类阶段必须**只读不写**，不允许修改任何模块文件。
> **铁律 §I.1.2-B**：一份文件只能落入一类（首匹配优先）；混合型文件（如理论 + 案例）按内容主体定，并在 `pending-extraction-queue.md` 备注。

### §I.1.3 阶段 3 · 提取（自动 / 分支化）

**目标**：把阶段 2 分类结果按对应分支抽取候选规律。

#### 分支 A · 理论提示词

```
抓取规则：
  - 抓所有铁律口诀（``` 代码块包裹的）
  - 抓所有表格（| ... | ... |）的标题行 + 全部数据行
  - 抓所有以『铁律』『口诀』『公式』『口决』开头的段落
  - 编号风格：保留原文（如 v2.0 的『第三步』『五律』；v2.1 的 Step 0/0.5/8）
输出：
  每条候选规律带『原文锚点：文件路径#段落起始关键词』
  写入 pending-extraction-queue.md 主表
```

#### 分支 B · 案例验证

```
抓取规则：
  - 抓『论断 X』『验证 X』『新规律 X』段落
  - 抓 ADD / REVISE / VERIFY 三类编号（统一改为 R-NNN-XXX-NN 风格，NNN = 案例号）
  - 抓事实型时间线（如『2018 戊戌年高考 701 分』）
输出：
  候选规律编号（R-NNN-ADD-NN / R-NNN-REVISE-NN / R-NNN-VERIFY-NN）
  来源案例 / 触发条件 / 拟入模块（A/B/C/D/E）
```

#### 分支 C · 合婚报告

```
抓取规则：
  - 抓 10 维评分表
  - 抓三窗口推荐（同居窗 / 领证窗 / 风险窗）
  - 抓『论断 X』推理链
输出：
  R-NNN-COMPAT-NN 编号
  拟入模块：D（合婚专属）+ B/C（部分跨模块）
```

#### 分支 D · 择日报告

```
抓取规则：
  - 抓七层递进打分
  - 抓三体系投票（盲派 1 / 子平 1 / 择吉 0.5）
  - 抓 5 维画像 + 大运前三步
输出：
  R-NNN-ZERI-NN 编号
  拟入模块：E（择日专属）
```

#### 分支 E · 高德臣 PDF

```
走 §I.2 PDF 子协议
当前沙箱无 PDF 解析能力 → 整份 PDF 进 pending-queue P0 待办
不允许凭印象编造 PDF 内容（参 §I.3 反幻觉硬约束）
```

### §I.1.4 阶段 4 · 候选入库（自动）

**目标**：把阶段 3 抽取的候选规律写入 `pending-extraction-queue.md` 主表。

每条候选规律必须填齐 8 栏（详见 `pending-extraction-queue.md §P.1`）：

```
| id | 来源文件 | 来源段落锚点 | 候选规律内容 | 拟入模块 | 候选等级 0.0-2.4 | 等待条件 | 入队日期 |
```

**候选等级初值规则**：

| 来源类型 | 初值（满分 2.4） | 说明 |
|---|---|---|
| 案例反馈 A 级 | 1.0 | 单案 A 级 = 1.0 / 2.4，差 1.4 才晋级 |
| 案例反馈 B 级 | 0.6 | 单案 B 级 |
| 案例反馈 C 级 | 0.3 | 单案 C 级 |
| 案例反馈 D 级 | 0.1 | 单案 D 级 |
| 理论提示词（已被引用） | 0.6 | 等同 B 级 |
| 理论提示词（未被引用） | 0.3 | 等同 C 级 |
| PDF 待 OCR | 0.0 | 内容未提取，等级为 0；提取后按本表重评 |

> **铁律 §I.1.4-A**：写入 `pending-extraction-queue.md` 时必须按入队日期倒序追加。
> **铁律 §I.1.4-B**：同一规律来自两个不同案例 / 文档时，**累加分数**而**非**新建条目（合并条目时在『来源文件』栏列出全部来源）。

### §I.1.5 阶段 5 · 晋级回写（人审 + 自动）

**目标**：累计加权 ≥ 2.4 → 该规律从 pending 移到对应模块文件 + `cases/<专题>.md` 规律分数板 + `blind-bazi-cases-index.md` 同步。

| Step | 动作 | 自动 / 人审 |
|---|---|---|
| 5-1 | 扫描 `pending-extraction-queue.md`，找出累计分 ≥ 2.4 的条目 | 自动 |
| 5-2 | 拟稿『写入哪个模块的哪一节』+『分数板增加哪一行』+『索引指纹库 / 案例索引追加哪行』 | 自动出方案 |
| 5-3 | 人审（用户授权 / 至少 1 个新案例验证）| **必须人审** |
| 5-4 | 真正写入主模块 + cases/<专题>.md + blind-bazi-cases-index.md | 半自动（按拟稿） |
| 5-5 | 更新 `META/source-trace.md` 三元组：来源 md / 段落锚点 / 当前模块:章节号 | 自动 |
| 5-6 | 重算 `META/module-coverage-matrix.md`，对应单元格从 ☐ 升 ◐ 或 ☑ | 自动 |
| 5-7 | 从 `pending-extraction-queue.md` 移除该条目（或标『已出队』）| 自动 |

> **铁律 §I.1.5-A**：阶段 5 不允许在『没有人审或没有新案例验证』的情况下执行。
> **铁律 §I.1.5-B**：阶段 5 完成后必须把所有四件套文件（matrix / queue / source-trace / 受影响的 cases-index）一次性同步提交，**禁止**只更新一两份造成不一致。

## §I.2 PDF 子协议（高德臣讲义专属）

> **背景**：本仓库 9 份高德臣 PDF 讲义合计约 470 页，是盲派命理体系的**理论母本**。
> **现状（2026-05-18）**：本沙箱网络模式 INTEGRATIONS_ONLY，无 PDF 解析能力（`pdftotext` / `pdfplumber` / `pypdf` / `pymupdf` 全部缺失，pip 被代理 403）。
> **结论**：当前 9 份 PDF 全部入 `pending-extraction-queue.md` 的 P0 待办；**禁止**凭印象编造高德臣讲义内容（参 v6.0.4 修复的『丑申合』伪规律扩散教训）。

### §I.2.1 PDF 入库前置条件

PDF 走完阶段 3（提取）需要满足以下任一条件：

| 条件 | 工具 | 备注 |
|---|---|---|
| 命令行 PDF 文本抽取 | `pdftotext` / `pdfplumber` / `pypdf` / `pymupdf` 任一 | 适用于文字型 PDF |
| OCR | `tesseract` + `pdf2image` / 在线 OCR API | 适用于扫描型 PDF |
| 人工转写 | 用户手动把章节转录为 md | 适用于含表格 / 图示的复杂讲义 |

> **铁律 §I.2-A**：上述工具任一可用前，PDF 阶段 3 必须**跳过**（不抽取内容），整份文件以 `0.0` 候选等级停在 `pending-extraction-queue.md` P0 区。
> **铁律 §I.2-B**：从用户对话或其它二手来源听来的 PDF『内容』**严禁**直接当作『来自高德臣讲义』入库；必须先有原文锚点（文件名 + 页码 + 段落首句）才能建立候选条目。
> **铁律 §I.2-C**：v6.0.4 修复的『丑申合』伪规律 = 反例标本 —— 它正是『五行相通被概念扩张为六合』的幻觉，本子协议直接对照其反向校准结果（详见 `module-a-paipan.md §A.1.4` 六条反幻觉铁律）。

## §I.4 与现有体系的闭环对接

```
┌─ 阶段 4 候选入库 ──→ pending-extraction-queue.md
│                            │
│                            ↓ 累计加权 ≥ 2.4
│                       (人审 + 新案例验证)
│                            │
│                            ↓
│      ┌─────────────────────┼─────────────────────┐
│      ↓                     ↓                     ↓
│  主模块文件         cases/<专题>.md         blind-bazi-cases-index.md
│  (A/B/C/D/E)         规律分数板 🟢          指纹库 + 索引 + 分数板
│      │                     │                     │
│      └─────────────────────┴─────────────────────┘
│                            │
│                            ↓
│                  META/source-trace.md（三元组同步）
│                            │
│                            ↓
└──────────────── META/module-coverage-matrix.md（重算覆盖度）

冲突情形 → rule-conflicts.md（系统性盲区或跨模块冲突）
星级偏差 → calibration-log.md（命中率回归）
```

### §I.4.1 与 `blind-bazi-cases-index.md` 的同步

- **指纹库**：阶段 5 涉及新案例 → 同步追加新指纹（日主-月令-格局-神煞）
- **案例索引**：新案例文件（如 `案例库/案例004_XX.md`）→ 在 §二案例索引按时间倒序追加
- **规律分数板**：阶段 4-5 涉及新规律 → 同步追加 / 升级该专题的规律行
- **§七 模块映射表**：阶段 5 写入主模块时 → 同步追加映射行

### §I.4.2 与 `rule-conflicts.md` 的同步

- 阶段 3 抽取出的候选规律若与已入库规律**矛盾** → 不直接入主模块，而是先进 `rule-conflicts.md` 争议池
- 抽取过程发现『系统性盲区』（如『模块 D 路由完全缺位』）→ 进 `rule-conflicts.md` 系统性盲区登记

### §I.4.3 与 `calibration-log.md` 的同步

- 阶段 5 写入的新规律若涉及星级预测（如 ★★★★★ 学历预测）→ 在 `calibration-log.md` 建立追踪行
- 后续案例验证 → 回写到 `calibration-log.md`，触发星级降级 / 升级

### §I.4.4 与 `META/source-trace.md` 的双向同步

- 阶段 5 每条新写入的规律 → 在 `source-trace.md` §S.1 追加三元组（来源 md / 段落锚点 / 当前模块:章节号）
- `source-trace.md` 与 `blind-bazi-cases-index.md` **双向同步**：`cases-index` 按案例视角组织，`source-trace` 按来源文档视角组织

## §I.5 人 + Skill 协同分工

| 阶段 | 模型自动 | 必须人审 |
|---|---|---|
| §I.1.1 识别 | ✅ 全自动（仅扫描 + 比对） | — |
| §I.1.2 分类 | ✅ 全自动 | — |
| §I.1.3 提取 | ✅ 自动抽取候选 | 不写入主模块（仅写 pending-queue） |
| §I.1.4 候选入库 | ✅ 自动写 pending-queue | — |
| §I.1.5 晋级回写 | ❌ 自动出方案 | ✅ **必须人审 + ≥ 1 个新案例验证** |
| PDF 抽取（§I.2）| ❌ 工具就绪前禁止 | ✅ 必须人审原文锚点后再入 pending |

> **铁律 §I.5-A**：模型在『候选 → 晋级』环节没有自动权 —— 哪怕 pending 累计已 ≥ 2.4，也必须等人审通过才走阶段 5。
> **铁律 §I.5-B**：人审通过后，模型可以**自动**完成阶段 5 的 5-4 / 5-5 / 5-6 / 5-7 四个写入动作（一次性完成，避免不一致）。

## §I.7 闭环演示（假想新加入 `案例库/案例004.md`）

> 用一个具体的演示走通五阶段，证明协议能闭环 —— 不需要外部代码支持。

**前提**：用户在 `案例库/` 下新加 `案例004_XX_YY_ZZ_AA.md`，内容是某 A 级回溯验证。

| 阶段 | 动作 | 输出 |
|---|---|---|
| **1 识别** | `glob 案例库/*.md` → 与矩阵 diff → 发现案例004 未入库 | 启动自检告警：『⚠ 检测到 1 份新案例未入库（案例004），是否触发 META 协议？』 |
| **2 分类** | 用户授权 → 路径在 `案例库/` 且文件名前缀为 `案例NNN_` → 落入『案例验证』类 | 走分支 B（抓 ADD/REVISE/VERIFY） |
| **3 提取** | 抓『论断 X』『新规律』段落，每条带原文锚点 | 候选规律 R-004-ADD-01 / R-004-VERIFY-01 等 |
| **4 候选入库** | 写入 `pending-extraction-queue.md`，初值 1.0/2.4（A 级） | pending 队列追加 N 条 |
| **5 晋级回写** | 人审通过 + 至少 1 个新案例佐证 → 写入对应模块（如 module-c-yunqi.md §C.X.X）+ cases/<专题>.md 分数板 + blind-bazi-cases-index.md 同步；source-trace.md 追加三元组；module-coverage-matrix.md 该行从 ☐ 升 ☑ | pending 出队 |

**演示终态**：5 件套（主模块 / cases / cases-index / source-trace / matrix）一次性同步完成，无遗漏；用户后续 `grep -n '案例004' .kiro/skills/` 能在多份文件中命中。
